        # QuickDebug工具

        TRD在M18平台上提供了一个隐藏工具： QuickDebug。

        这个工具用来对系统指定App的所有editor进行如下测试：打开、读取单据、保存单据，对所有EBI执行如下测试：打开、读取数据。

        这个工具的初衷是TRD在发布新的版本前，检查新版本是否存在明显BUG。现在对所有开发者以及测试人员开放，让大家可以在需要时进行批量操作。

        # 启用QuickDebug

        只需要在login页面，或者app首页时，在浏览器地址栏加上参数?quickDebug=true

        附带这个参数后，主页右上方会多出一个工具按钮

        点击这个按钮，就会启动QuickDebug功能。

        # 配置QuickDebug功能

        启动QuickDebug后，会弹出配置Dialog

        Debug Option说明：

        • Test Read Action For Module : 是否测试Module的数据读取功能
        • Test Save Action For Module: 是否测试Module的数据保存功能
        • Test Refresh Action For EBI: 是否测试EBI的数据读取功能
        • Show Error Only： 在显示测试结果时，只显示发生错误的editor/ebi
        • Batch Size & Duration： 每批次测试的editor数量， 当一个批次完成后，系统会暂停Duration指定的时间(毫秒ms)，以便浏览器释放资源。 这两个选项仅仅在测试机内存空间不足的时候使用。 BatchSize=0（默认）意味着不分批次测试，所有editor在一个批次内完成测试。
        • Debug Range用来选择需要测试的范围，以“App“作为测试单元，可以勾选需要测试的App。 App是以”开发者“来进行分类的。
        • Include UDF module: 指没有归属APP的module，多为用户自定义的Module.

        # 开发者为特定页面指定测试内容

        TRD只提供了统一的测试内容。 如果开发者要在QuickDebug时，为某个页面指定单独的测试动作，请按照以下步骤：（所有开发，在js文件中进行）

        1. 监听QuickDebug动作，增加测试动作

          $(document).on('quickDebug', function(event, info, setting) {
	if(quickDebug) {
		quickDebug.addAction({code: 'testCode', description: 'test description'}, 
         function() {
          	//测试代码
         });
	}
});

        2. 测试代码

          一般情况下，测试代码是一系列ajax请求组成。系统会自动根据ajax的请求数量以及返回状态来判断测试代码是否执行完成，以及执行的结果（出现ajax error则意味着失败）。

          开发者也可以自行追踪测试代码的完成情况和结果，如果需要自行追踪，则在测试代码的最前方，添加以下代码:

          var activeAction = quickDebug.getActiveAction();
if(activeAction) {
  activeAction.manualDone = false;
}
//测试代码
//activeAction.manualDone = true; //当测试完成时执行。

          注意 如果测试代码不包含ajax， 并且存在异步调用的Javascript代码，则必须自行追踪测试代码完成情况。

        3. 示例：

          $(document).on('quickDebug', function(event, info, setting) {
  if(quickDebug) {
    if(setting.action.moduleRead) {
      //测试读取动作
      if($('button[data-action="readFirst"]').length > 0) {
        quickDebug.addAction({code: 'read', description: 'Read first Record'}, function() {
          $('button[data-action="readFirst"]').click();
        });
      }
    }

    if(setting.action.ebiRefresh) {
      //测试EBI Refresh功能，由于测试代码存在异步JavaScript代码，所以要自行追踪测试状态
      if($('.analysisToolbar #commonTable .refresh').length > 0) {
        quickDebug.addAction({code: 'ebiRefresh', description: 'Refresh EBI'}, function() {
          var activeAction = quickDebug.getActiveAction();
          if(activeAction) {
            activeAction.manualDone = false;
          }
          $(document).on('ebiDataLoaded', function(event, status) {
            var activeAction = quickDebug.getActiveAction();
            if(activeAction) {
              activeAction.manualDone = true;
              activeAction.status = (status ? 'success' : 'failed');
            }
          });
          $('.analysisToolbar #commonTable .refresh').click();
        });
      }
    }
  }

        # 测试用WebServices

        # Ebi Mail Sender Test WS

        1. 使用Mail Sender数据测试。

          测试人员必须先准备好测试数据，在模块【发送EBI邮件]模块建立一个有效的数据，确定数据id.

          Path: ebiMail/testMailJob

          Method： GET

          Params:

          ​ jobId: long类型， 有效的[发送EBI邮件]模块的数据id

          Response:

          ​ 如果status=ok，则意味着Mail Sender Test通过

          ​ 如果status=failed，意味着Mail Senhder Test不通过。

          1. 使用EBI Format数据测试。

        测试人员必须先准备好测试数据，在EBI模块建立一个有效的Format，确定Format id.

        Path: ebiMail/testEbiMail

        Method： GET

        Params:

         `formatId`:  `long`类型， 有效的EBI模块Format `id`

        Response:

        ​ 如果status=ok，则意味着Mail Sender Test通过

        ​ 如果status=failed，意味着Mail Senhder Test不通过。
